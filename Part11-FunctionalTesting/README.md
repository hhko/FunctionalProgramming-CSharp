# Part 11 — Functional Testing (효과·전문성 테스트)

## 11부의 배경

기초부터 우리는 추상을 정의할 때마다 법칙을 함께 말했습니다. Functor 의 항등 / 합성, Applicative 의 다섯 법칙, Monad 의 좌·우 항등과 결합 — 이 법칙들은 "추상이 자연스럽게 작동함" 을 보장하는 계약입니다. 그리고 Part 1 에서 익힌 최소 property 검증으로, 각 추상은 자기 장 안에서 그 법칙을 임의 입력으로 그 자리에서 확인했습니다. **법칙 검증은 이미 각 장에서 끝납니다.**

11부의 출발점은 그 다음입니다. 기초의 순수 코드로는 닿지 않는, **늦은 기계장치가 필요한 테스트** 입니다. 효과 (`Eff<RT>` / `IO`), 동시성, 스트리밍, 자원 수명은 부수 효과와 시간과 비결정성이 얽혀 있어, 검증에 전용 도구 (테스트 런타임 / 테스트 더블) 가 필요합니다.

**부수 효과를 값으로 인코딩하고 런타임만 바꿔 끼우면 효과 코드마저 결정적으로 테스트할 수 있다는 것** — 그것이 11부의 핵심이자 함수형 테스트의 전문 영역입니다. 책 목표 5 (테스트 자동화의 자신감) 의 정면입니다.

- 효과 코드의 결정적 테스트 — `Eff<RT>` 의 런타임 `RT` 에 테스트 더블 (`MemoryConsole` / `MemorySystemEnvironment`) 을 주입해, IO 가 섞인 코드를 실제 부수 효과 없이 검사.
- 동시 · 스트리밍 · 자원 효과 테스트 — 비결정성과 시간을 길들여 STM / 스트림 / Schedule / Resource 를 검사.
- property-based 심화 — Part 1 의 최소 도구를 무작위 생성기 + 축소 (shrinking) 로 키우고, 실무 도구 (CsCheck / FsCheck) 로 이행.

## 11부의 목표

11부를 마치면 독자는 다음을 갖게 됩니다.

- `Eff<RT>` 효과 코드를 테스트 런타임 (`MemoryConsole` 등 테스트 더블) 으로 결정적·빠르게 검사.
- 동시성 (STM / Atom) · 스트리밍 (StreamT / Pipes) · 자원 (Schedule / Resource) 효과의 비결정성을 길들여 테스트.
- Part 1 의 최소 property 검증을 입력 생성기 (generator) 와 축소 (shrinking) 로 심화하고, CsCheck / FsCheck 류 도구로 이행.
- 기초 각 장에 흩어진 법칙 검증을 "어떤 `F` 든 받는 법칙 모듈" 로 일반화하고, `Has<RT, Trait>` DI 로 테스트 용이한 설계를 짜기.

부수 효과가 섞인 효과 코드를 결정적으로 검증하는 능력이 11부의 도달점입니다.

## 11부의 무대 — 늦은 기계장치가 필요한 테스트

| 대상 | 검증의 어려움 | 도구 |
|---|---|---|
| **효과 코드** (`Eff<RT>`) | 부수 효과 · 외부 의존성 | `Sys.Test` 런타임 + `MemoryConsole` 테스트 더블 |
| **동시 · 스트리밍 · 자원** | 비결정성 · 시간 · 수명 | 결정적 스케줄 · 골든 / 종료 테스트 |
| **임의 입력 성질** (심화) | 무한한 입력 공간 | 생성기 + 축소 (CsCheck / FsCheck) |

순수 함수는 같은 입력에 같은 출력이라 기초의 각 장에서 이미 property 로 법칙을 검증했습니다. 11부는 그 위에서, **효과는 값으로 인코딩되어 런타임만 바꿔 테스트 더블을 주입할 수 있다** 는 설계를 검증으로 거둡니다. 기초 ~ 10부의 모든 효과 설계가 여기서 테스트로 이어집니다.

## 11부의 학습 흐름

| 장 | 두 축에서의 자리 | 한 줄 |
|---|---|---|
| 36장 | 효과를 결정적으로 | 테스트 런타임 / `MemoryConsole` 로 `Eff<RT>` 검사 |
| 37장 | 비결정성을 길들이다 | 동시 · 스트리밍 · 자원 효과 테스트 |
| 38장 | 성질을 끝까지 | property-based 심화 + 함수형 테스트 아키텍처 |

## 3개 장의 구성

### 36장 — 효과 코드의 결정적 테스트

`Eff<RT>` 의 핵심 (런타임 `RT` 가 의존성이라는 점) 을 테스트에 활용합니다. 운영 런타임 대신 `MemoryConsole` / `MemorySystemEnvironment` 같은 테스트 더블을 주입하면, 콘솔·파일·시간이 섞인 효과 코드를 실제 부수 효과 없이 결정적으로 검사할 수 있습니다. 7부의 `Has<RT, Trait>` DI 설계가 테스트 용이성으로 이어지는 자리입니다. LanguageExt v5 의 `Sys.Test` 런타임이 그 실체입니다.

### 37장 — 동시 · 스트리밍 · 자원 효과 테스트

효과 코드의 가장 까다로운 영역인 비결정성을 길들입니다. STM / Atom 의 원자성을 결정적으로 검증하고, StreamT / Pipes 파이프라인을 골든 / 종료 테스트로 확인하며, Schedule (재시도 횟수) 과 Resource (개폐 순서) 의 수명을 검사합니다. 9 ~ 10부에서 만든 동시 · 스트리밍 효과가 테스트로 이어지는 자리입니다.

### 38장 — property-based 심화 + 함수형 테스트 아키텍처

Part 1 에서 도입한 최소 property 검증 (`ForAll`) 을 본격적으로 키웁니다. 무작위 생성기 (generator) 로 입력을 만들고, 실패하면 축소 (shrinking) 로 최소 반례를 찾는 발상을 직접 구현한 뒤 CsCheck / FsCheck 류 도구로 옮깁니다. 기초 각 장에 흩어진 법칙 검증을 "어떤 `F : Functor<F>` 든 받는 법칙 모듈" 로 일반화하고, `Has<RT, Trait>` DI 로 테스트 용이한 코드를 설계하는 함수형 테스트 아키텍처로 마무리합니다.

## 11부의 코드

본문 예제는 모두 `code/Part11-FunctionalTesting/` 에 실행 가능한 형태로 들어 있습니다 (외부 패키지 의존 0). 효과 테스트 더블은 v5 `Sys.Test` 런타임을 기반으로 하며, property 엔진 (생성기 + `ForAll` + shrinking) 은 의존성 0 으로 직접 구현합니다. 장 재편에 맞춰 코드 디렉토리도 새 구성 (효과 → 동시·스트리밍 → property 심화) 으로 정렬됩니다.

이 책은 직접 구현 축에 맞춰 테스트 더블 · property 엔진의 메커니즘을 의존성 0 으로 만들어 봅니다. 실무에서는 동일 개념을 **xUnit + Shouldly** (단언), **CsCheck / FsCheck** (property-based) 로 옮깁니다 — 각 `Tests/*.cs` 의 `bool` 헬퍼를 `[Fact]` + `ShouldBe…` 로 감싸면 그대로 표준 테스트가 됩니다.

## 11부의 진입점

10부를 마쳤다면 36장 — 효과 코드의 결정적 테스트 부터 시작합니다. 법칙 검증은 이미 기초 ~ 3부의 각 장에서 property 로 끝났으므로, 11부는 효과 코드를 결정적으로 다루는 전문 테스트로 곧장 들어갑니다.
